# 2者間ゲーム実験仕様書（動的更新版）

## 1. 目的
2者間ゲーム（囚人のジレンマ、スタグハント、チキンゲーム、およびベースラインとしてのハーモニーゲーム）における意思決定と反応時間（RT）を測定する実験プログラムの作成。
最終的には以前作成したEEM（jsPsychタスク）のコードと統合し、Qualtrics上で動作させる（埋め込む）ことを想定。

## 2. 開発ワークフロー（進捗ステータス）
- [x] **ステップ1:** ローカルでの基本構造作成（`experiment.html` / `main.js` / `style.css`の作成）
- [x] **ステップ2:** Qualtricsへの実装準備（データ保存処理等）
- [x] **ステップ3:** 既存課題（EEM・SVO）との統合（`EEM_SD`リポジトリで運用）

## 3.1 タスクフロー（EEM統合版）
統合後の全体フローは以下の通り進行します。
1. **【課題1】EEM課題**（教示 → 理解度クイズ → 練習試行 → 本番試行）
2. **【課題2】SVO課題**（マウスによる9段階スライダー選択）
3. **【課題3】SD課題: 教示**（ルールの説明、ポイントの表の見方）
4. **【課題3】SD課題: 理解度クイズ**（2問：自分とBさんのポイントの組み合わせを選択）
5. **【課題3】SD課題: 練習試行**（3回）
6. **【課題3】SD課題: 本番試行**（全38試行を13, 13, 12の3ブロックに分割・間に休憩あり）
7. **終了画面**（Qualtrics移行処理・フルスクリーン解除）

## 4. 各要素の詳細要件

### 4.1 教示画面
* 基本的に**すべてマウスのクリック操作（ボタン）**で進行する。
* 「協力」「非協力」など価値判断を含むラベルは絶対に使用しない。
* 教示の冒頭で、「実験で獲得した最終的なポイントに応じて、参加報酬とは別に追加のボーナス報酬が支払われる」旨を明確に説明する。
* **相手の呼称の区別:** 課題の性質が異なることを明確にするため、EEM・SVO課題での相手は「Aさん」、SD課題での相手は新たにマッチングされた「Bさん」と表記する。
* SD教示の後半で「ポイントの表」を画面に出して以下を説明：
  * あなたとBさんの選択が交差したマスが結果となる
  * 自分の選択によってBさんのポイントが変わり、Bさんの選択によって自分のポイントが変わる
  * それぞれの選択によるポイントの変化の具体例を、ハイライトを用いて1ページずつ論理的かつ具体的に説明する。
  * **マッチングの透明性:** 事後マッチングという表現は避け、「ペアとなる相手は完全に匿名であり、実験中に結果は画面上に提示されない。あなたの選択は、ランダムに選ばれた参加者の選択と照らし合わせられ、お互いのボーナスに反映される」という事実ベースの説明を行う。
  * **事後マッチングと報酬決定:** 教示内に「参加者全員から回答を回収した後、別の参加者をランダムにマッチングし、実際の選択に基づいて報酬額を決定する」「50回のうち1回が選ばれて計算される」ことを明記し、3ページに分けて読みやすく提示する。
  * **教示用ダミー利得パターン:** 教示やクイズで使用する表のポイントは、本番でのアンカリングを防ぐため、本番で使用しない特殊な数値の組み合わせ（定数7、倍率3を用いた 69, 57, 45, 33）を使用し、本番の計算構造を類推しにくくする。
### 4.2 SD課題の理解度クイズ
* キーボード入力（本番）と混同しないよう、**マウスのクリック（画面上のボタン）**で解答させる。
* 計2問実施する。
  * 提示された表に対して、「あなた: X pt, Bさん: Y pt」と「あなた: Y pt, Bさん: X pt」の2択から正しい組み合わせを選択させる。
* **ループ制限:** 間違えた場合は再挑戦ループに戻るが、**各問題最大2回まで**とする。2回間違えた場合は正解を提示して強制的に次に進める。

### 4.3 試行フロー・時間パラメータ
* クイズ終了後、本番のキーボード操作環境を強制するため**マウスカーソルを非表示**にする。
* 練習課題の開始画面で画像を用いて「これ以降はキーボード（F/Jキー）を使用する」旨を明確に説明し、スペースキーで練習試行を開始させる（※制限時間の表示は行わない）。
1. **注視点 (Fixation):** 1000ms
2. **ブランク画面:** 500ms
3. **ポイントの表の提示（キー入力待ち）:**
   * 画面中央にマトリックス形式のポイントの表を提示。
   * 上部に「どの選択肢を選びますか？」という案内文を表示。
   * あなたの選択: 「[F]キー」「[J]キー」
   * Bさんの選択: 「パターン1」「パターン2」
   * 表内の数値にはすべて「pt」の単位を付与する。
   * キーボードの「F」または「J」が入力されるまで待機（タイムアウト制限30秒）。
   * **時間切れ処理:** 30秒以内に選択されなかった場合、画面中央に落ち着いた教示風UIで「制限時間切れ」と2.5秒間提示し、強制的に次の試行へ進む（未選択として記録）。
4. **選択のハイライト（500ms）:**
   * 選択した列（Fなら左、Jなら右）の背景色を薄い青色（#e7f1ff 等）に変更し、500ms提示する（案内文は透明化してレイアウトを維持）。
5. **試行間ブランク:** 500ms
* **ブロックごとの休憩:** 38試行を13, 13, 12に分け、間に「少し休憩をとってください」という画面を挟む（スペースキーで再開）。

### 4.4 利得表のUIと視覚多様性への配慮
* 数値の色付けに依存しないよう、各セル内で**「あなた」「Bさん」というラベルの直下に黒字で数値を表示**する形式を採用し、視覚的多様性に完全対応する。
* 13インチモニターでも見やすいよう、**全体の幅を960px程度に拡大**し、文字サイズを大きく確保。
* **画面中央への完全配置:** 表の左側ヘッダー（パターン1など）をCSSで表枠外の絶対配置にすることで、純粋なデータセルの境界線の交差点が完全な画面中央（X:50%, Y:50%）に来るよう配置する。

## 5. 項目水準（条件）とデータ構造

* **ゲームの種類:**
  * 囚人のジレンマ (PD)
  * スタグハント (SH)
  * チキンゲーム (CH)
  * ハーモニーゲーム (HG: アテンションチェック用)

* **データ構造と計算ロジック:**
  数値をベタ書きせず、以下の計算式とパラメータで、すべてが2桁の利得になるように動的に生成する。
  1. **計算式:** `(ベース + 定数) × 倍率`
  2. **ベース配列:** `const base = [16, 12, 8, 4]` （等間隔4）
  3. **定数と倍率のパターン (4種類):**
     定数 `10, 15` と 倍率 `2, 3` を組み合わせ、以下の4パターンを定義する。
     - P2: (C:10, M:2) => `[52, 44, 36, 28]`
     - P3: (C:10, M:3) => `[78, 66, 54, 42]`
     - P5: (C:15, M:2) => `[62, 54, 46, 38]`
     - P6: (C:15, M:3) => `[93, 81, 69, 57]`
  4. **組み合わせ構造（1ゲームにつき6パターン）:**
     - 対称パターン (4個): 自分と相手が同じパターンを使用（例: P2 vs P2）。
     - 非対称パターン (2個): 定数10を使用し、倍率2と3を組み合わせる。
       - 有利: 自分 P3(C:10, M:3) vs 相手 P2(C:10, M:2)
       - 不利: 自分 P2(C:10, M:2) vs 相手 P3(C:10, M:3)
  5. **試行数と順序構造:**
     - 6パターン × 3ゲーム（PD, SH, CH） = 18条件。
     - それぞれについて左右の表示位置を入れ替えて2回繰り返す = 36試行。
     - これにアテンションチェック用のHG（対称パターン）を左右1回ずつ＝2試行追加し、**合計38試行**とする。
     - **固定ランダムシード化:** 参加者間での順序効果を統制するため、完全ランダムではなく固定のシャッフル配列を用意して全参加者に同一の順序で提示する。
     - **第1・第38試行のアンカリング:** 
       - 第1試行には「最も基本となる対称PDパターン（P2: C:10, M:2）」を固定配置する。
       - 第38試行には、第1試行と「全く同じ利得で左右反転させた試行」を配置する。
       - 目的: 全ての文脈を経験する前（第1試行）と後（第38試行）での協力行動の変容（Pre-Post）を測定するため。
